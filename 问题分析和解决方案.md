# 📋 Content 数据库状态问题分析与解决方案

## 🔍 问题现象

从数据库查询结果看，content 表中很多数据都是 **DRAFT 状态**，但预期应该有 **PENDING、APPROVED、REJECTED、ONLINE、OFFLINE** 等其他状态的数据。

## 🎯 根本原因分析

### 1. **测试数据生成方式不当**

`测试数据生成脚本.sql` 中直接通过 SQL INSERT 语句插入各种状态的数据：

```sql
-- 直接插入 PENDING 状态
INSERT INTO content (title, content, status, creator_id, created_at, updated_at)
VALUES ('数据库性能优化技巧', '...', 'PENDING', 3, NOW(), NOW());

-- 直接插入 APPROVED 状态
INSERT INTO content (title, content, status, creator_id, created_at, updated_at)
VALUES ('微服务架构设计', '...', 'APPROVED', 2, NOW(), NOW());

-- 直接插入 ONLINE 状态
INSERT INTO content (title, content, status, creator_id, created_at, updated_at)
VALUES ('Git 工作流最佳实践', '...', 'ONLINE', 2, NOW(), NOW());
```

### 2. **业务流程与数据流程不一致**

根据代码分析，内容状态转换有严格的业务流程控制：

- **创建** → 自动设置为 `DRAFT`
- **提交审核** → 必须从 `DRAFT` 或 `REJECTED` 转到 `PENDING`
- **审核** → 从 `PENDING` 转到 `APPROVED` 或 `REJECTED`
- **发布** → 必须从 `APPROVED` 转到 `ONLINE`
- **下线** → 从 `ONLINE` 转到 `OFFLINE`

这些转换在各 Service 实现中有完整的 **权限和状态校验**：

```java
// ContentServiceImpl.java - line 118-121
if (content.getStatus() != ContentStatus.DRAFT && 
    content.getStatus() != ContentStatus.REJECTED) {
    throw new BusinessException(400, "只有草稿或拒绝状态的内容才能提交审核");
}
```

### 3. **直接 SQL 插入绕过了业务逻辑**

直接执行 SQL 脚本可以：
- ✗ 绕过权限校验
- ✗ 绕过状态转换的合法性检查
- ✗ 创建不符合业务流程的数据状态

### 4. **可能的场景**

- **场景1**: 脚本执行但没有成功（网络问题、脚本错误）
- **场景2**: 数据库后来被重置（只保留了 data.sql 的初始数据）
- **场景3**: 脚本反复执行导致只有最后执行的成功
- **场景4**: 用户界面没有正确调用相关 API 来转换状态

## ✅ 解决方案

### 方案 A：通过 API 调用正确生成测试数据（推荐）

创建一个新的数据生成脚本，通过 HTTP API 调用来生成符合业务流程的测试数据。

**步骤**:
1. 创建内容（API 自动设置为 DRAFT）
2. 提交审核（DRAFT → PENDING）
3. 审核通过（PENDING → APPROVED）
4. 发布上线（APPROVED → ONLINE）
5. 下线（ONLINE → OFFLINE）

### 方案 B：增强 SQL 脚本的合法性

修改 `测试数据生成脚本.sql`，确保：
1. 只在必要时直接插入状态数据
2. 添加详细的注释说明为什么直接插入
3. 提供一个 API 调用版本作为主要方案

### 方案 C：添加数据库视图和校验

创建一个触发器或存储过程来验证状态转换的合法性。
