# 数据与日志系统设计文档

## 一、统计指标定义说明

### 1.1 内容统计指标

| 指标名称 | 含义 | 统计字段 |
|---------|------|---------|
| 草稿数（DRAFT） | 正在编写中、未提交的内容数 | content.status |
| 待审核数（PENDING） | 已提交等待审核的内容数 | content.status |
| 已通过数（APPROVED） | 审核通过的内容数，代表"发布量" | content.status |
| 已拒绝数（REJECTED） | 审核未通过的内容数 | content.status |
| 已上线数（ONLINE） | 已发布上线的内容数 | content.status |
| 已下线数（OFFLINE） | 已下线的内容数 | content.status |
| 时间范围发布量 | 指定时间段内通过审核的内容数 | content.status + content.created_at |

### 1.2 审核统计指标

| 指标名称 | 含义 | 计算方式 |
|---------|------|---------|
| 审核总数 | audit_record 总条数 | COUNT(*) |
| 通过数 | 状态为 APPROVED 的审核记录数 | COUNT(status='APPROVED') |
| 通过率 | 审核通过的比例 | 通过数 / 总数 × 100% |

## 二、统计实现方案

### 2.1 技术选型

使用 **Spring Data JPA 方法命名约定** 生成查询，无需手写 SQL：

```java
// 按状态统计 → 自动生成 SELECT COUNT(*) FROM content WHERE status = ?
long countByStatus(ContentStatus status);

// 按状态+时间范围统计 → 自动生成带 BETWEEN 的 SQL
long countByStatusAndCreatedAtBetween(ContentStatus status, LocalDateTime start, LocalDateTime end);
```

### 2.2 接口设计

| 接口 | 方法 | 说明 | 返回示例 |
|------|------|------|---------|
| `/api/stats/summary` | GET | 全局统计汇总 | 状态分布 + 审核通过率 |
| `/api/stats/publish` | GET | 按时间统计发布量 | 需传 startDate/endDate 参数 |

**`/api/stats/summary` 返回结构：**
```json
{
  "contentStatusDistribution": {
    "DRAFT": 3,
    "PENDING": 1,
    "APPROVED": 10,
    "REJECTED": 2,
    "ONLINE": 8,
    "OFFLINE": 1
  },
  "auditPassRate": {
    "total": 12,
    "approved": 10,
    "passRate": 83.33
  }
}
```

**`/api/stats/publish` 返回结构：**
```json
{
  "startDate": "2026-04-01",
  "endDate": "2026-04-26",
  "publishedCount": 5
}
```

## 三、日志系统设计方案

### 3.1 操作日志（数据库持久化）

记录用户的每一次关键业务操作，存入 `operation_log` 表。

**表结构：**

| 字段 | 类型 | 说明 |
|-----|------|------|
| id | BIGINT | 主键，自增 |
| user_id | BIGINT | 操作人ID |
| action | VARCHAR(50) | 操作类型（见下表） |
| target_id | BIGINT | 操作对象ID（内容ID/用户ID等） |
| created_at | DATETIME | 操作时间（自动记录） |

**已覆盖的操作类型：**

| action 值 | 触发时机 |
|-----------|---------|
| CREATE_CONTENT | 创建内容 |
| UPDATE_CONTENT | 编辑内容 |
| DELETE_CONTENT | 删除内容 |
| SUBMIT_REVIEW | 提交审核 |
| PUBLISH_CONTENT | 发布内容 |
| OFFLINE_CONTENT | 下线内容 |
| APPROVE_CONTENT | 审核通过 |
| REJECT_CONTENT | 审核拒绝 |
| CREATE_USER | 创建用户 |
| DELETE_USER | 删除用户 |
| CREATE_PERMISSION | 创建权限 |

### 3.2 系统日志（SLF4J 输出）

在 Service 层添加 SLF4J 日志，按以下级别使用：

| 级别 | 使用场景 | 示例 |
|-----|---------|------|
| INFO | 关键操作成功 | "内容创建成功：contentId=5" |
| WARN | 权限不足、参数有误 | "创建内容权限不足：userId=3" |
| DEBUG | 过程性中间信息 | JWT token 验证过程 |
| ERROR | 系统异常 | 数据库连接失败 |

## 四、数据库优化策略

### 4.1 索引设计

在高频查询字段上建立索引，通过 JPA `@Table(indexes=...)` 注解声明，Spring 启动时自动创建。

**content 表：**

| 索引名 | 字段 | 用途 |
|-------|------|------|
| idx_content_status | status | 按状态统计/过滤内容 |
| idx_content_creator_id | creator_id | 按创建者查询内容 |
| idx_content_created_at | created_at | 按时间范围统计发布量 |
| idx_content_status_created_at | status, created_at | 时间+状态组合查询（最常用） |

**audit_record 表：**

| 索引名 | 字段 | 用途 |
|-------|------|------|
| idx_audit_status | status | 统计审核通过率 |
| idx_audit_reviewer_id | reviewer_id | 按审核员查询 |
| idx_audit_created_at | created_at | 按时间查询审核记录 |

**operation_log 表：**

| 索引名 | 字段 | 用途 |
|-------|------|------|
| idx_log_user_id | user_id | 查询某用户的操作记录 |
| idx_log_action | action | 按操作类型查询 |
| idx_log_created_at | created_at | 按时间查询操作记录 |

### 4.2 分页查询优化

**问题**：原代码用 `findAll()` 取出所有数据再在内存里过滤分页，数据量大时严重影响性能。

**解决**：改用 Spring Data JPA 的 `Pageable`，让数据库只返回当前页的数据：

```java
// 修改前（问题代码）
List<Content> all = contentRepository.findAll();       // 全量加载

// 修改后（正确做法）
Pageable pageable = PageRequest.of(page-1, pageSize, Sort.by("createdAt").descending());
Page<Content> page = contentRepository.findByConditions(status, creatorId, pageable); // 只返回一页
```

## 五、系统增强说明

### 5.1 新增统计模块

- 新增 `StatisticsService`：提供内容状态分布、审核通过率、时间维度发布量三类统计
- 新增 `StatisticsController`：对外暴露 `/api/stats/summary` 和 `/api/stats/publish` 接口

### 5.2 日志输出增强

以下 Service 新增了 SLF4J 日志：

- `ContentService`：记录内容创建、权限不足等关键节点
- `JwtAuthenticationFilter`：记录 token 接收、验证结果，便于排查认证问题
- `AuthenticationService`：记录登录成功/失败及原因

### 5.3 已知问题修复

| 问题 | 修复方案 |
|------|---------|
| 分页接口用 `findAll()` 全量加载 | 改用 JPA `Pageable` |
| 单元测试断言硬编码 ID | 改为按 username 断言，避免自增 ID 不稳定 |
| **密码明文存储** | 引入 `BCryptPasswordEncoder`，登录时用 `matches()` 比对，数据库中只存哈希值，原始密码不可逆还原 |
| **无单元测试** | 新增 `ContentServiceTest`（5个）、`AuditServiceTest`（3个）、`StatisticsServiceTest`（3个），使用 Mockito 进行纯单元测试；3个分页集成测试（H2内存库）；共15个测试用例 |
| **缺少认证机制** | 引入 JWT 认证：登录接口返回 `accessToken`，后续请求通过 `Authorization: Bearer <token>` 携带，由 `JwtAuthenticationFilter` 过滤器统一校验，无法通过 Header 伪造用户身份 |
| **API 不完整（无分页）** | 用户列表、权限列表改为分页接口，新增 `PaginationRequest` / `PaginationResponse` 统一封装，支持页码、每页条数、排序字段、排序方向参数 |
