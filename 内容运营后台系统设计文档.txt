1.业务场景拆解
内容平台整体业务流程如下：
运营人员创建内容 → 提交审核 → 审核员审核 → 审核通过后上线 → 后续可下线
角色职责：
管理员：系统配置。用户管理，角色管理，权限分配。
运营人员：创建内容，修改内容，提交审核。
审核员：内容审核。查看待审核内容，执行通过/驳回，并填写驳回原因。


2.核心流程设计
2.1业务流程图

2.2内容状态流转图

DRAFT → PENDING → APPROVED → ONLINE → OFFLINE
   ↑       ↓
      ← REJECTED 

生命周期流程：
DRAFT（草稿）：运营创建内容后的初始状态
PENDING（待审核）：运营提交审核后进入该状态
APPROVED（审核通过）：审核员审核通过后进入该状态
REJECTED（审核拒绝）：审核员审核未通过，返回修改
ONLINE（已上线）：审核通过后由运营执行上线操作
OFFLINE（已下线）：运营可对已上线内容进行下线


2.3 审核流程设计
在内容进入审核阶段后，由审核员对内容进行审核处理。审核员可以执行以下操作：
（1）审核通过：内容状态由 PENDING 变为 APPROVED，之后可由运营人员执行上线操作；
（2）审核驳回：内容状态变为 REJECTED，并返回给运营人员进行修改，修改后可重新提交审核；
系统通过状态流转控制审核流程，确保内容在未审核通过前无法进入上线状态。

2.4 异常情况处理逻辑
在实际业务流程中，可能出现以下异常情况，需要进行约束与处理：
（1）非法状态流转
问题：内容在不合法状态下执行操作 (如：DRAFT 状态直接上线) 
处理方式： 
系统校验当前状态，仅允许合法流转 
非法操作直接拒绝，并返回错误提示 
（2）重复提交审核
问题：内容已处于 PENDING 状态再次提交 
处理方式： 
限制重复提交 
返回“当前内容正在审核中”提示 
（3）越权操作
问题：普通用户执行审核或上线操作 
处理方式： 
进行权限校验（基于角色） 
无权限操作直接拒绝 
（4）审核结果冲突
问题：同一内容被多次审核或重复操作 
处理方式： 
审核记录只允许处理一次 
后续操作需校验审核状态 
（5）数据不存在或已删除
问题：操作的内容不存在 
处理方式： 
返回资源不存在错误（404） 
阻止后续流程执行 


3.数据库表结构设计
3.1 设计说明
本系统采用关系型数据库进行数据存储，围绕内容管理与审核流程设计核心数据模型，包括用户表、角色表、权限表、内容表、审核记录表以及操作日志表。

3.2 核心数据表设计
3.2.1 用户表（user）
表结构：
字段名	类型	说明
id	bigint	用户ID（主键）
username	varchar	用户名
password	varchar	密码
role_id	bigint	角色ID(外键,关联role.id)
created_at	datetime	创建时间
说明：
用于存储系统用户信息，并通过 role_id 关联角色表，实现用户角色划分。

3.2.2 角色表（role）
表结构：
字段名	类型	说明
id	bigint	角色ID（主键）
role_name	varchar	角色名称（admin / reviewer / operator）
说明：用于定义系统中的角色类型，不同角色对应不同的操作权限。

3.2.3 权限表（permission）
表结构：
字段名	类型	说明
id	bigint	权限ID（主键）
permission_name	varchar	权限名称（如 content:create）
说明：用于定义系统中可执行的操作权限。
权限模型说明：
系统基于 RBAC（Role-Based Access Control）模型，实现用户、角色与权限之间的解耦。用户通过角色关联权限，从而实现灵活的权限控制机制。

3.2.4 内容表（content）
表结构：
字段名	类型	说明
id	bigint	内容ID（主键）
title	varchar	标题
content	text	内容
status	varchar	内容状态
creator_id	bigint	创建人ID（外键，关联 user.id）
created_at	datetime	创建时间
updated_at	datetime	更新时间

内容状态定义：
DRAFT（草稿） 
PENDING（待审核） 
APPROVED（审核通过） 
REJECTED（审核拒绝） 
ONLINE（已上线） 
OFFLINE（已下线） 
说明：
status 字段用于表示内容当前所处状态，实现内容从创建、审核到上线与下线的完整生命周期管理。

3.2.5 审核记录表（audit_record）
表结构：
字段名	类型	说明
id	bigint	主键
content_id	bigint	内容ID（外键，关联 content.id）
reviewer_id	bigint	审核人ID（外键，关联 user.id）
status	varchar	审核结果
comment	varchar	审核意见
created_at	datetime	审核时间

审核状态定义：
APPROVED（通过） 
REJECTED（拒绝） 
说明：
用于记录每一次审核操作，实现审核留痕与可追溯性，支持多次审核场景。

3.2.6 操作日志表（operation_log）
表结构：
字段名	类型	说明
id	bigint	主键
user_id	bigint	操作人ID（外键，关联 user.id）
action	varchar	操作行为
target_id	bigint	操作对象ID
created_at	datetime	操作时间

说明：
用于记录系统关键操作行为，包括内容创建、审核、上线及下线等操作，提升系统的可审计性与问题追踪能力。

3.3 关键设计说明
（1）状态字段设计
系统中内容状态与审核状态均采用枚举值进行定义，确保状态值的规范性与一致性，避免字符串不统一带来的问题，同时便于业务逻辑控制。
（2）内容状态与审核状态区分
content.status 表示内容的整体状态，而 audit_record.status 表示单次审核操作的结果，两者分别用于不同层级的状态管理。
（3）时间字段设计
各核心数据表均包含时间字段（created_at、updated_at），用于记录数据变化过程，便于系统追踪与数据分析。

4.系统架构设计
4.1 架构概述
本系统采用分层架构设计，整体后端分为以下模块：


4.2 模块划分
模块	职责
权限模块	管理用户角色与权限，提供权限校验接口，控制用户访问范围
内容模块	提供内容创建、修改、查询功能，实现内容生命周期管理
审核模块	管理内容审核流程，记录审核结果，实现通过/驳回及回流逻辑

4.3 分层结构
Controller(接口层):对外提供 RESTful 接口，接收请求、返回数据和错误码
Repository(DAO):封装对数据库的访问，包括增删改查操作，提供给 Service 层使用
Service(业务逻辑层):实现核心业务逻辑，如审核流程、权限校验、内容状态管理
Entity(数据实体层):映射数据库表结构，定义数据字段和类型

5. 接口设计（Controller 层）
5.1 接口规范
接口风格：RESTful 
请求格式：JSON 
返回格式： 
{
  "code": 200,
  "message": "success",
  "data": {...}
}
错误码示例： 
code	message	说明
400	Bad Request	参数错误
401	Unauthorized	未授权或权限不足
404	Not Found	资源不存在
500	Internal Error	系统内部错误

5.2 核心接口列表（示例）
5.2.1创建内容接口
URL: /api/content/create 
方法: POST 
功能：创建内容，初始状态为 DRAFT
请求参数（Controller 接收）： 
{
  "title": "string",
  "content": "string",
  "creator_id": 123
}
返回示例： 
{
  "code": 200,
  "message": "success",
  "data": {
    "content_id": 1,
    "status": "DRAFT"
  }
}

5.2.2 提交审核接口
URL: /api/content/submit 
方法: POST 
功能：提交内容进入审核流程（DRAFT → PENDING）
请求参数： 
{
  "content_id": 1,
  "operator_id": 123
}
返回示例： 
{
  "code": 200,
  "message": "success",
  "data": {
    "status": "PENDING"
  }
}

5.2.3 审核处理接口
URL: /api/content/audit 
方法: POST 
功能：审核内容，通过或驳回
请求参数： 
{
  "content_id": 1,
  "reviewer_id": 456,
  "action": "APPROVED",  // APPROVED / REJECTED
  "comment": "审核意见"
}
返回示例： 
{
  "code": 200,
  "message": "success",
  "data": {
    "status": "APPROVED"
  }
}
5.2.4 内容查询接口
URL: /api/content/list
方法: GET 
功能：按条件查询内容
请求参数： 
{
  "status": "PENDING",   // 可选：DRAFT / PENDING / APPROVED / REJECTED / ONLINE / OFFLINE
  "creator_id": 123      // 可选
}
返回示例： 
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "content_id": 1,
      "title": "示例内容",
      "status": "PENDING",
      "creator_id": 123
    }
  ]
}

5.2.5 内容上下线接口
URL: /api/content/publish 或 /api/content/offline 
方法: POST 
功能：控制内容上线或下线
请求参数： 
{
  "content_id": 1,
  "operator_id": 123
}
返回示例： 
{
  "code": 200,
  "message": "success",
  "data": {
    "status": "ONLINE"  // 或 OFFLINE
  }
}
